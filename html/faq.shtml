1. What is exmap?

    Exmap is a tool which allows you to see how much physical memory is in use by different process, mapped file, ELF sections and ELF symbols at a given moment in time.

    In particular, an attempt is made to accurately account for shared memory in the following way; when a page of physical memory is found to be shared between N processes, each process is given a 1/N share of that page.

    Exmap doesn't allow you to see details on how and where memory on the heap is allocated. Tools such as memprof and possibly valgrind are more use in this case.

  2. On which platforms does exmap run?

    Exmap is linux-specific, since it uses a linux kernel loadable module. Additionally, the kernel module requires a recent kernel (2.6.11 or higher) in order to successfully compile or run.

    Exmap has been reported to run successfully on Ubuntu 5.10 (Breezy Badger) and Fedora Core 4.

    Previous versions (0.5.1 and lower) produce warnings on some systems with a 64-bit kernel. More recent versions should work.

    Please let me know if you have problems running on a particular platform or can add to the list of known working platforms above.

  3. What do I need to build exmap?

    You'll need the basic development tools such as gcc, make etc. Additionally, since part of exmap is a loadable kernel module, you'll need the kernel headers for your current kernel installed.

    On Ubuntu, you can arrange this with:

    sudo apt-get install build-essential linux-kernel-headers

  4. What do I need to run exmap?

    The userspace tool which interprets the data from the kernel module is written in perl, so you'll need perl installed. The UI is written using the gtk+ toolkit (which underlies GNOME) so you'll need both that toolkit and the perl bindings.

    On Ubuntu, you can arrange this with:

    sudo apt-get install libgtk2-perl

  5. Exmap just hangs on startup, chewing 100% CPU!

    Sorry. Taking and analyzing the snapshot can take a while. On my 2.8Ghz P4 with 512Mbyte, this can take around 16 seconds of 100% CPU. (Down from 30s for release 0.5.1 :-)

    I'm looking at how to improve this. If this bugs you or makes it unusable for you, please let me know so I can judge how important this problem is.

  6. At startup I get lots of warnings like: "Can't load info for pid XXX at Exmap.pm line YYY"

    You're probably running Exmap as root. This is probably fine, the pids should be kernel threads (with names in square brackets, like [kswapd]. These kernel threads don't have an associated VM area, so it can't be loaded.

    If neither of this is the case, please let me know as its probably a bug.
